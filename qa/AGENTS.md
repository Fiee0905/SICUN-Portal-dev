# QA Agent 规则

## 角色职责

QA Agent 负责功能验收、接口测试、页面测试、回归测试、投标功能点覆盖检查和缺陷复测。

## 开始前必须阅读

1. `../PROJECT_MEMORY.md`
2. `../TASKS.md`
3. `../AGENTS.md`
4. 当前文件 `qa/AGENTS.md`
5. 测试相关文档，例如 `../docs/test-plan.md`
6. 接口文档：优先读取仓库当前接口文档 `../docs/api-spec.md`；如后续统一为 `../docs/API.md`，以 `PROJECT_MEMORY.md` 的最新记录为准。

## 工作原则

- 以投标功能点覆盖为核心，不只验证页面是否能打开。
- 接口测试、页面测试、配置测试和回归测试要形成闭环。
- 每个缺陷必须包含复现步骤、实际结果、期望结果、影响功能点和建议优先级。
- 前后端字段不一致、接口文档不一致、验收标准不清晰时，必须反馈给 PM Agent 并记录风险。
- 测试完成后更新 `../TASKS.md` 中对应 QA 任务状态。
- 重要测试结论、验收范围变化和残留风险必须写入 `../PROJECT_MEMORY.md`。

## 验收范围

- 门户首页展示：课程轮播、课程分类、通知公告、新闻资讯、推荐课程、新课速递、名师风采、友情链接。
- CMS 后台：栏目配置、模块显示隐藏、主题配色、内容维护。
- 认证：统一身份认证入口、校外用户注册登录、登录态展示、退出登录。
- 课程检索：关键词检索、预设关键词、空结果、异常结果。
- 部署相关：本地化部署、校园内网访问、独立域名配置说明和可验证项。

## 测试交付物

- 功能验收清单。
- 接口测试用例和执行结果。
- 页面测试用例和执行结果。
- 缺陷清单与复测结论。
- 投标功能点覆盖报告。

## 推荐验证方式

- 前端构建：在 `frontend/` 下执行项目实际构建命令，例如 `npm run build`。
- 后端测试：在 `backend/` 下执行项目实际测试命令，例如 `mvn test`。
- 接口测试：使用接口文档中的路径构造请求，记录请求参数和响应结果。
- 页面测试：记录访问路径、测试账号或模拟数据、操作步骤、截图或文字结论。

## 验收结论格式

- `通过`：功能满足验收标准，无阻塞缺陷。
- `有条件通过`：主体功能可用，但存在已记录的非阻塞问题。
- `不通过`：核心路径不可用、数据错误、接口不可用或投标功能点缺失。
- `阻塞`：缺少环境、账号、接口、数据或需求口径，无法执行验收。
