| 可测 | 已有明确页面或接口路径，环境启动后可执行验收。 |
| 部分可测 | 核心路径可测，但真实环境、账号、权限、数据或联调仍缺口。 |

## Cross-Feature Evidence Requirements

| 证据类型 | 最低要求 |
| --- | --- |
| 页面证据 | 首页、课程页、登录/注册页、CMS 配置页至少保留访问路径、操作步骤、结果截图或文字记录。 |
| 接口证据 | 每个涉及接口至少记录 method/path、请求参数、响应状态、响应体关键字段、trace/request id。 |
| 数据证据 | CMS 配置、用户状态、课程状态、文章状态变更需记录数据库或接口查询前后结果。 |
| 权限证据 | 匿名、普通用户、CMS 编辑、审核员、超级管理员至少覆盖允许和拒绝两类结果。 |
| 缺陷证据 | 每个不通过或阻塞项需记录复现步骤、实际结果、预期结果、影响功能点、建议优先级。 |

## Auth And Permission Addendum

专项验收清单见 `qa/AUTH_PERMISSION_ACCEPTANCE.md`。该清单围绕注册登录、role 权限、课程 permission、后台 CMS 权限和删除旧版 CMS 的影响，定义了手工测试矩阵、阻塞项和 Go/No-Go 标准。

当前 QA 判定：

| 范围 | QA 结论 |
| --- | --- |
| 注册登录 | `/login`、`/register` 和认证 API 可做页面/API 冒烟；正式验收需密码哈希、用户状态、token、refresh/logout 和 CAS 校验闭环。 |
| role/permission | `/users/me` 可用来核对角色和权限声明；后台接口权限拦截未确认前，任何成功访问 `/admin/*` 都不能视为权限通过。 |
| 课程 permission | `GET /portal/courses`、详情过滤和 `POST /portal/courses/{id}/launch` 均需按匿名/校外/校内/管理员矩阵复测；launch 当前应未登录 401、无权 403、有权返回跳转信息。 |
| 后台 CMS 权限 | CMS 栏目、文章、门户配置、轮播、快捷入口等写接口必须按权限码返回 `401/403/200`；前端菜单隐藏不替代后端拒绝。 |
| 旧版 CMS 删除影响 | 删除或收敛 `/cms/*` 前必须验证旧入口跳转/拒绝、无敏感 mock 数据、无死链、无权限绕过，且 `/admin/*` 配置闭环仍可用。 |
