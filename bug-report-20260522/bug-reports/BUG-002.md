# BUG-002: 后端运行路径仍硬编码 localhost:7001 外部 CAS/LMS 地址

复核时间：2026-05-21T13:03:01Z
复核人：qa-team/bug-reviewer
测试工作区：/mnt/d/work2.0/SICUN-Portal_test
后端 API：http://172.19.192.1:8080/api/v1

## 1. 复核对象

- 接口/路径：
  - GET /api/v1/auth/cas/login-url?redirectUri=...
  - POST /api/v1/portal/courses/{id}/launch 在 course.launchUrl 为空时的 fallback
- 缺陷候选：qa/bug-reports/automation-smoke-defect-candidates.md 中 “P2 - Hard-coded localhost:7001 external URLs remain in backend runtime paths”
- 证据文件：
  - qa/evidence/no_mock_scan_summary.txt
  - qa/evidence/api_permission_smoke_result.json
- 相关代码只读定位：
  - backend/src/main/java/com/edu/portal/auth/controller/AuthController.java:29-32
  - backend/src/main/java/com/edu/portal/portal/controller/PortalController.java:126-129

## 2. 复核结论

成立。

运行时 GET /auth/cas/login-url 返回的 loginUrl 包含 http://localhost:7001/cas/login；源码扫描同时确认课程 launch fallback 中存在 http://localhost:7001/course-platform/.../launch。

该问题属于环境配置/部署可用性缺陷。与 BUG-001 相比，它不是直接认证绕过，但会导致非本机访问、联调环境、验收环境或生产环境中的 CAS/LMS 跳转地址错误。

## 3. 复现路径

### 3.1 CAS login-url 运行时复现

执行：

```bash
curl -i 'http://172.19.192.1:8080/api/v1/auth/cas/login-url?redirectUri=http%3A%2F%2Fqa.local%2Fcallback'
```

复核实测结果：

```text
HTTP/1.1 200
Content-Type: application/json

{"code":0,"message":"success","data":{"loginUrl":"http://localhost:7001/cas/login?service=http://qa.local/callback"},...}
```

### 3.2 LMS launch fallback 代码证据

qa/evidence/no_mock_scan_summary.txt 记录：

```text
[localhost_external_placeholder] backend/src/main/java/com/edu/portal/auth/controller/AuthController.java:31 String loginUrl = "http://localhost:7001/cas/login?service=" + redirectUri;
[localhost_external_placeholder] backend/src/main/java/com/edu/portal/portal/controller/PortalController.java:128 : "http://localhost:7001/course-platform/courses/" + course.getId() + "/launch";

源码只读复核结果与上述一致：

String loginUrl = "http://localhost:7001/cas/login?service=" + redirectUri;
```

```java
String launchUrl = course.getLaunchUrl() != null
        ? course.getLaunchUrl()

```

## 4. 实际结果 vs 预期结果

实际结果：

- CAS 登录 URL 在后端响应中写死为 localhost:7001。
- 课程 launchUrl 为空时，后端会生成 localhost:7001 的 LMS fallback。
- 对浏览器/客户端而言，localhost 指向客户端本机，而不是部署服务器或真实 CAS/LMS 服务。

预期结果：

- CAS/LMS 外部地址应来自环境变量或 profile 配置。
- 不同环境（local/test/staging/prod）应配置各自的 CAS baseUrl、LMS baseUrl。
- 非 local profile 不应默认回退到 localhost。
- 缺失必要外部地址时应 fail fast 或返回明确配置错误，而不是返回不可用的 localhost 链接。

## 5. 严重级别和优先级

- 严重级别：P2
- 优先级：高，建议在验收前修复；若验收范围包含真实 CAS/LMS 跳转，则升级为阻塞项

评级依据：

- 影响 CAS 跳转和课程平台跳转，是重要集成路径。
- 有明确绕行方案：数据库为课程维护正确 launchUrl、或运行环境刚好是本机 7001；但这不适用于正式联调/验收。
- 不直接导致数据破坏或权限绕过，因此低于 BUG-001。

## 6. 影响范围

已确认影响：

- CAS 登录入口 URL 生成。
- 课程 launchUrl 为空时的默认跳转。
- WSL/远端/局域网/容器/生产环境中的外部系统链接可达性。

潜在影响：

- 前端用户点击 CAS 登录后跳到错误主机。
- 课程 launch fallback 跳到用户本机 localhost，导致无法打开课程平台。
- 验收环境中外部系统集成失败。

## 7. 是否阻塞验收

有条件阻塞。

- 若最终验收要求 CAS/LMS 集成链路可用：阻塞验收。
- 若本轮验收只覆盖公开门户、基础登录和已有课程 launchUrl，且明确不验收真实 CAS/LMS 跳转：不单独阻塞，但必须作为验收风险项记录，并在部署前修复。

考虑 BUG-001 已经阻塞认证验收，整体发布/验收仍应阻塞。

## 8. 修复建议 / 进一步验证建议

修复建议：

1. 将 CAS baseUrl、LMS/course platform baseUrl 移入配置，例如 application.yml + 环境变量。
2. local profile 可默认 localhost；test/staging/prod profile 不允许默认 localhost。
3. 对 redirectUri/service 做 URL 编码与白名单/域名校验，避免开放重定向风险。
4. 当课程 launchUrl 为空且 lms.base-url 未配置时，返回明确配置错误或业务错误，不生成 localhost fallback。
5. 增加配置扫描和接口回归：非 local profile 响应中不得包含 localhost:7001。

进一步验证建议：

- 配置测试环境 CAS_BASE_URL、LMS_BASE_URL 后，调用 /auth/cas/login-url 确认返回目标域名。
- 准备 launchUrl 为空的课程数据，登录后调用 /portal/courses/{id}/launch，确认 fallback 使用配置域名或明确报错。
- 对 no-mock/no-localhost 扫描结果设为 CI 质量门禁。
