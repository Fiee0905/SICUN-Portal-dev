# BUG-001: CAS callback 接受无效 ticket 并签发内部用户 token

复核时间：2026-05-21T13:03:01Z
复核人：qa-team/bug-reviewer
测试工作区：/mnt/d/work2.0/SICUN-Portal_test
后端 API：http://172.19.192.1:8080/api/v1

## 1. 复核对象

- 接口：POST /api/v1/auth/cas/callback
- 缺陷候选：qa/bug-reports/automation-smoke-defect-candidates.md 中 “P1 - CAS callback accepts invalid ticket and issues an internal-user token”
- 证据文件：
  - qa/evidence/cas_callback_probe.txt
  - qa/evidence/no_mock_scan_summary.txt
- 相关代码只读定位：
  - backend/src/main/java/com/edu/portal/auth/controller/AuthController.java:35-38

## 2. 复核结论

成立。

该接口在收到明显无效的 CAS ticket 时返回 HTTP 200，并签发 teacher001 / internal 用户的 accessToken、refreshToken。代码层面也显示 casCallback 未校验 request.ticket 和 request.service，直接调用 userService.authenticate("teacher001", "123456")。

这不是单纯测试脚本误报；automation 与 manual-tester 的发现均被复核复现，且运行时行为与源码证据一致。

## 3. 复现路径

前置：后端服务运行于 http://172.19.192.1:8080/api/v1。

执行：

```bash
curl -i -X POST 'http://172.19.192.1:8080/api/v1/auth/cas/callback' \
  -H 'Content-Type: application/json' \
  -d '{"ticket":"INVALID_QA_REVIEW_20260521","service":"http://qa.local/callback"}'
```

复核实测结果：

```text
HTTP/1.1 200
Content-Type: application/json

{"code":0,"message":"success","data":{"accessToken":"dev.Mzp0ZWFjaGVyMDAxOmludGVybmFs...","refreshToken":"dev.Mzp0ZWFjaGVyMDAxOmludGVybmFs...","expiresIn":7200,"user":{"id":3,"username":"teacher001","displayName":"任课教师","role":"internal","userType":"TEACHER","source":"CAS","status":"ACTIVE"...}}}
```

已有证据 qa/evidence/cas_callback_probe.txt 也记录：

```text
POST /auth/cas/callback invalid ticket
HTTP 200
... "username":"teacher001" ... "role":"internal" ... "source":"CAS" ...
```

## 4. 实际结果 vs 预期结果

实际结果：

- 无效 ticket 被接受。
- 接口返回 HTTP 200 / code=0。
- 系统签发 teacher001 的 internal accessToken 和 refreshToken。
- ticket/service 未参与真实 CAS 票据校验。

预期结果：

- 无效 CAS ticket 必须被拒绝。
- 不应签发任何用户 token。
- 建议返回 HTTP 400/401 或业务错误码，例如 INVALID_CAS_TICKET。
- 只有在服务端向 CAS 服务端完成 ticket validation，且 service 与票据匹配后，才可映射/创建/登录对应用户。

## 5. 严重级别和优先级

- 严重级别：P1
- 优先级：最高，发布/验收前必须修复

评级依据：

- 该问题属于认证绕过/测试 mock 逻辑泄露到运行时。
- 任意调用方无需有效 CAS 认证即可获取内部教师账号 token。
- token 可访问所有 internal 角色授权的接口，影响核心身份认证链路。
- 当前权限烟测显示未登录后台接口有 401、普通用户访问后台有 403，说明系统依赖 token/role 做访问控制；因此伪造 internal token 会扩大到内部用户可访问范围。

备注：若该系统目标环境明确仍为纯本地演示且不会进入任何联调/验收环境，可临时降低发布风险；但按当前“最终验收”口径，不应放行。

## 6. 影响范围

已确认影响：

- CAS 登录回调接口。
- 身份认证完整性。
- internal/TEACHER 用户权限边界。

潜在影响：

- 课程 launch、个人中心、内部可见课程、教师/内部角色相关业务。
- 所有依赖 AuthSession token 的后续接口。
- 安全验收、生产准入、外部系统 CAS 联调。

## 7. 是否阻塞验收

阻塞验收。

原因：这是核心认证链路缺陷。无效凭据可换取有效内部用户 token，属于严重回归/安全缺陷。即使其他公开 API 与基础权限烟测通过，也不能覆盖该认证绕过风险。

## 8. 修复建议 / 进一步验证建议

修复建议：

1. 移除 casCallback 中的硬编码 teacher001/123456 登录逻辑。
2. 接入 CAS ticket validation：服务端使用 request.ticket + request.service 调用 CAS /serviceValidate 或同等接口。
3. 校验 CAS 返回主体、service 绑定关系、用户唯一标识、失败码。
4. CAS 校验失败时返回 401/400，禁止签发 token。
5. 若必须保留本地演示逻辑，应仅允许在显式 local/mock profile 下启用，并且默认关闭；验收/生产 profile 必须 fail closed。
6. 增加自动化回归用例：
   - invalid ticket => 非 200 或 code != 0，且无 accessToken。
   - 空 ticket/service => 参数校验失败。
   - ticket 与 service 不匹配 => 拒绝。
   - valid CAS ticket => 正常登录并映射正确用户。

进一步验证建议：

- 修复后重复执行上述 curl。
- 解码/检查返回体，确认无 accessToken/refreshToken。
- 使用内部受保护接口验证无效 ticket 不能获得任何授权访问能力。
- 对所有 profile 配置做 no-mock/no-hardcoded-account 扫描。
