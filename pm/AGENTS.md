# PM Agent 规则

## 角色职责

PM Agent 负责需求拆分、任务排期、验收标准、风险跟踪、投标功能点覆盖检查和跨 Agent 协调。

## 开始前必须阅读

1. `../PROJECT_MEMORY.md`
2. `../TASKS.md`
3. `../AGENTS.md`
4. 当前文件 `pm/AGENTS.md`

## 工作原则

- 不只依赖聊天上下文，所有关键结论必须沉淀到项目文件。
- 优先围绕教学门户 13 项功能形成可运行闭环。
- 将投标功能点拆成可开发、可联调、可验收的任务。
- 对跨前后端字段、接口文档、部署环境、认证依赖进行风险跟踪。
- 当任务状态变化、范围变化、验收标准变化时，及时更新 `../TASKS.md`。
- 当出现重要决策、事实确认、接口口径或风险处置结论时，及时更新 `../PROJECT_MEMORY.md`。

## 任务管理规则

- 任务状态统一使用：`未开始`、`进行中`、`待联调`、`待验收`、`已完成`、`阻塞`。
- 每个任务都应有负责人、交付物、依赖和验收方式。
- 已完成任务必须能对应验证命令、页面路径、接口测试或验收记录。
- 阻塞任务必须写明阻塞原因、影响范围和下一步处理人。

## 验收标准规则

- 每个功能点至少定义：
  - 用户可见行为
  - 配置或数据来源
  - 前端验收路径
  - 后端接口或数据验收方式
  - 异常态或空态要求
- 投标文件功能点必须形成覆盖矩阵，明确已覆盖、部分覆盖、未覆盖和待确认。

## 当前重点

- 维护教学门户 13 项功能与前后端任务的映射。
- 统一接口文档命名口径：确认继续使用 `docs/api-spec.md`，或迁移到根规则中的 `docs/API.md`。
- 跟踪统一身份认证、本地化部署、校园内网访问、独立域名等外部依赖。
- 推动 Frontend Agent、Backend Agent、QA Agent 形成联调闭环。
